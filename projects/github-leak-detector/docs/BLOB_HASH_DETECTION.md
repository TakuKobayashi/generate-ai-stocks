# Blobハッシュ検索の技術詳細

## 概要

GitHub Leak Detectorは、Gitの内部で使用されるファイル内容のハッシュ値（blob SHA-1）を使用してソースコードの流出を検出します。これは最も正確で信頼性の高い検出方法です。

## なぜBlobハッシュなのか？

### コミットハッシュとの比較

| 特徴 | コミットハッシュ | Blobハッシュ |
|-----|----------------|-------------|
| **検索可能性** | GitHub APIで制限あり | Code Search APIで検索可能 |
| **改ざん耐性** | 書き換え可能（git rebase） | ファイル内容が同じなら不変 |
| **検出範囲** | コミット履歴全体が必要 | 個別ファイル単位で検出 |
| **精度** | 高い | 非常に高い |

### パターン検索との比較

| 特徴 | パターン検索 | Blobハッシュ検索 |
|-----|------------|----------------|
| **精度** | 中（誤検知あり） | 非常に高い |
| **ファイル名変更** | 影響を受ける | 影響されない |
| **コード整形** | 影響を受ける | 影響されない |
| **言語依存** | ある程度依存 | 完全に非依存 |
| **検出速度** | 速い | 中程度 |

## Blobハッシュの仕組み

### Gitにおけるblobハッシュ

Gitは各ファイルの内容をblobオブジェクトとして保存し、SHA-1ハッシュで識別します。

```bash
# ファイルのblobハッシュを確認
git hash-object README.md

# すべてのファイルのblobハッシュを確認
git ls-files -s
```

### ハッシュの計算方法

同じ内容のファイルは常に同じハッシュ値を持ちます。

## 検出アルゴリズム

### 1. Blobハッシュの抽出
### 2. バッチ検索（複数のハッシュをORで連結）
### 3. マッチングカウント
### 4. マッチング率の計算
### 5. リスクレベルの判定

## パフォーマンス最適化

### バッチサイズの選択

```bash
# 小規模プロジェクト
--blob-batch-size 10

# 中規模プロジェクト（デフォルト）
--blob-batch-size 20

# 大規模プロジェクト
--blob-batch-size 30
```

## 閾値設定のガイドライン

### 推奨設定

```bash
# 厳格（確実な流出のみ）
--match-threshold 80

# バランス（推奨デフォルト）
--match-threshold 30

# 敏感（疑わしいものも含める）
--match-threshold 10
```

## まとめ

Blobハッシュ検索は最も推奨される検出方法です:

✅ **高精度**: ファイル内容の完全一致を検出
✅ **改ざん耐性**: コミット履歴の書き換えに影響されない
✅ **言語非依存**: あらゆるプロジェクトで使用可能
✅ **柔軟な閾値**: マッチング率で段階的に判定
✅ **効率的**: バッチ検索でAPI制限を最適化
